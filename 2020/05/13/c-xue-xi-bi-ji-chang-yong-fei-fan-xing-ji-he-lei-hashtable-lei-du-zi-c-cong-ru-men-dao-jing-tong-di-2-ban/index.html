HashTable类

HashTable称为哈希表，利用键/值来存储数据。每个元素都是一个键/值对，并且是一一对应的，通过“键”就可以得到“值”。

常用属性
| 属性名称 | 属性说明 |
|–|–|
| Count | 获取包含在Hashtable中键/值对的数目 |
| Keys | 获取包含在Hashtable中所有键的集合 |
| Values | 获取包含在Hashtable中所有值得集合 |
常用方法
| 方法名称 | 方法说明 |
|–|–|
| Add | 将带有指定键和值得元素添加到Hashtable中 |
| Clear | 从Hashtable中移除所有元素 |
| Contains | 确认Hashtable是否包含特定键 |
| GetEnumberator | 返回IDictionaryEnumberator，可以遍历Hashtable |
| Remove | 从Hashtable中移除带有指定键的元素 |
常用构造函数声明

Hashtable ht1 = new Hashtable(); // 使用所有默认值构建哈希表实例（初始容量为 16，负载因子为 0.75），实际容量为16 * 0.75 = 12
Hashtable ht2 = new Hashtable(20); // 指定哈希表的初始容量为20，实际容量为20 * 0.75 = 15
Hashtable ht3 = new Hashtable(20, 0.8f); // 初始容量为20个元素，加载因子为0.8,实际容量为10 *0.8 = 16
Hashtable ht4 = new Hashtable(s1); // 传入实现了IDicitionary接口的参数创建哈希表

加载因子：

  • 加载因子是表示Hsah表中元素的填满的程度。若:加载因子越大,填满的元素越多,好处是,空间利用率高了,但冲突的机会加大了。反之,加载因子越小,填满的元素越少。好处是:冲突的机会减小了,但空间浪费多了
  • 冲突的机会越大,则查找的成本越高.反之,查找的成本越小.因而,查找时间就越小

示例：

Hashtable openWith = new Hashtable();
openWith.Add("bmp", "paint.exe");
openWith.Add("did", "paint.exe");
openWith.Add("rtf", "wordpad.exe");
Console.WriteLine("键=\"rtf\"，值={0}", openWith["rtf"]); // wordpad.exe

openWith["doc"] = "winword.exe"; // 如果对不存在的键设置值，则添加新的键值对

// 通常添加之前用ContainsKey来判断某个键是否存在
if(!openWith.ContainsKey("ht"))
{
    openWith.Add("ht", "ht.exe");
    Console.WriteLine(openWith["ht"]); // ht.exe
}

// 遍历
foreach(DictionaryEntry de in openWith) // Hashtable的键/值是DicitionaryEntry类型
{
    Console.WriteLine("键 = {0}，值 = {1}", de.Key, de.Value);
}
/**
 * 键 = ht，值 = ht.exe
 * 键 = rtf，值 = wordpad.exe
 * 键 = bmp，值 = paint.exe
 * 键 = did，值 = paint.exe
 * 键 = doc，值 = winword.exe
 */

// 删除
openWith.Remove("doc");
if (!openWith.ContainsKey("doc"))
    Console.WriteLine("false"); // false
Console.ReadKey();

遍历时应注意，Hashtable的每个元素都是一个键/值对，因此元素类型既不是键的类型，也不是值的类型，而是DictionaryEntry类型。
